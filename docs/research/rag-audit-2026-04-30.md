# RAG Audit — analysis pipeline + chat (2026-04-30)

Полный аудит analysis-пайплайна и чат-сервиса с предложениями по улучшениям по трём осям: **качество**, **цена**, **скорость**. Все цифры основаны на production-данных за период 25–29 апреля 2026.

---

## Cost baseline (production, last 4 days)

```
Chat — Gemini Pro:    avg 32k input + 1.9k output = $0.088/turn  (€0.074, ₽7.92)
Chat — Gemini Lite:   avg 19k input + 0.6k output = $0.006/turn  (€0.005, ₽0.55)
                      Pro is 14× more expensive than Lite.
                      Currently ~44% Pro / 56% Lite split — ought to be ~20/80.

Analysis (Lite only): avg $0.276/book (₽25), range $0.07-$0.60
                      LLM 76% / Embeddings 24%
                      1 of 8 runs failed (12%) = real money lost
                      Avg duration: 3 min, range 1-8 min
```

**Где сидят 19k input на чат-турне (Lite avg):**
- System prompt + правила: ~2-3k токенов
- История диалога (compacted summary + последние пары): 1-4k
- **RAG evidence pack: 10-14k (≈70% input)** — biggest cost driver
- Вопрос пользователя: tiny

---

## Quick frame — две находки в начале

**1. RAG evidence pack — это не фиксированный pre-baked блок**, а markdown, отрендеренный `search_paragraphs` (иногда `search_scenes` + `read_passages`) внутри tool-loop. Правильные рычаги стоимости:
- Размер tool-результата
- Количество tool-call'ов
- Дедупликация

См. `apps/web/src/lib/bookChatService.ts:9244-9269` — sections concatenated: continuous slices ≤18k chars + evidence groups + paragraph hits ≤900 chars × top-K. Один и тот же параграф часто появляется и в "Evidence groups", и в "Параграф-хиты" — двойная стоимость.

**2. ~3000 строк мёртвого кода в `bookChatService.ts`** (12k всего). `buildChatPreplan` / `compileEvidencePack` / `createCompiledEvidenceAnswerSystemPrompt` / `synthesizeCompiledFallbackAnswer` (lines 3960, 6052, 6605, 6724-6817) **никогда не вызываются** из live-пути. В production живёт только `planBookChatExecution` + `streamBookChatAnswer`. Это объясняет почему файл такой страшный — половина не работает.

Live путь: `route.ts` → `streamBookChatThreadReply` → `streamBookChatAnswer` → `planBookChatExecution` (LLM #1) → `streamText` с `EVIDENCE_TOOL_CHAT_TOOL_NAMES = ["search_scenes","search_paragraphs","read_passages"]` (LLM #2, может быть multi-step).

### 2. Drop the duplicated paragraph in `search_paragraphs` markdown sections

**Где:** `bookChatService.ts:9244-9269`
**Effort:** S

**Mechanism:** Когда `search_paragraphs` возвращает результат, markdown содержит `Continuous slices` + `Evidence groups` + `Параграф-хиты`. **Risk:** Несколько тестов могут snapshot'ить markdown. Resnapshot.

### 3. Tighten `prepareStep` — relax forced-tool gate after step 0

**Где:** `bookChatService.ts:8890-8908, 11866`
**Effort:** S

**Mechanism:** Сейчас step 0 — `toolChoice: "required"` всегда когда планнер сказал `toolPolicy=required`. В сочетании с `MAX_AUTONOMOUS_TOOL_STEPS=8` LLM продолжает выпускать follow-up `read_passages` даже после чистого `search_paragraphs`. Backlog T3: `required` слишком агрессивен. Конкретный diff:
- Держать `required` только когда планнер emitted ≥2 `queryGroups` (multi-part question) ИЛИ `complexity ≠ simple`
- Иначе использовать `"auto"` со step 0

Не нужна новая heuristic — планнер уже produced `queryGroups` и `entityHints`.

**Impact:** Маленький-но-чистый quality lift на small-talk / trivial-fact misclassified как "book question" (модель может ответить без поиска → меньше hallucinations from over-filtering). Cost: −10-15% unnecessary tool calls. Sense check: при 44% Pro / 56% Lite split, даже Lite турны avg 19k input в основном из tool results.

**Risk:** Модель может skip retrieval на borderline case; `toolPolicy=auto` + system prompt уже говорит "если по книге — ОБЯЗАТЕЛЬНО вызови инструменты", покрытие сохраняется.

### 5. Make planner output strict about `searchQueries` quality

**Где:** `bookChatService.ts:3375-3416`
**Effort:** S

**Mechanism:** Текущий planner prompt просит 5-10 queries per multi-part question. На практике он дампит giant flat list, и search tool гоняет ОДИН и тот же query четырежды из-за dedup-on-text-only. Добавить:
- `searchQueries.length` ≤ 3 для simple, ≤ 6 для medium, ≤ 8 для hard
- Требовать каждый `queryGroups[i].part` иметь **distinct anchors** (планнер позволено доказать через `notes` строку)

Это prompt-engineering, не heuristic.

**Impact:** Меньше near-duplicate retrieval calls → меньше rerank calls → и quality (less chapter-distribution skew toward over-queried region) и cost. Embedding cost — 24% analysis но также matters per-chat-turn (один embed per query). С ranking enabled, каждый retrieval также гоняет Vertex Ranking call (`bookChatService.ts:1014-1132`) — складывается.

**Risk:** None — чисто prompt edit. Добавить single regression test.

**Effort:** S

**Mechanism:** Compaction injects summary как `role: "assistant"`. Anti-mimicry rules пытаются защитить, но несколько jailbreak transcripts в тестах всё ещё treat'ят это как previous reply. **Impact:** Закрывает residual jailbreak channel который уже tracked в тестах; minor token win (no XML wrapper duplication).

**Risk:** Нужно re-check `replaceLastUserMessageContent` чтобы оба prefix coexist в одном user message.

### 7. Stop duplicating `<scene>` map когда планнер says `complexity=simple`

**Где:** `bookChatService.ts:8961-9012, 11833`
**Effort:** S

**Mechanism:** `search_scenes` *advertised* в system prompt каждый turn даже когда планнер правильно классифицировал "simple book fact". Модель часто вызывает его всё равно как "warm-up". Привязать `search_scenes` registration к `decision.modelTier === "pro" || (queryGroups.length >= 2)`. Иначе omit tool из `createEvidenceToolChatTools` output полностью — модель буквально не может pick wrong one.

**Impact:** Прямой fix для T1 (planner picks wrong tool ~43%). На ~50% Lite турнов (simple facts) дропаем wasted tool call: один embedding call + один rerank call + ~1-2k input токенов scene markdown. Estimated savings ₽0.05/turn на Lite (почти half of current Lite cost) + small latency cut.

**Risk:** Маленькое количество "where is the scene with X" вопросов могут пострадать; планнер уже routes их в `complexity=hard` так что мы держим `search_scenes` available там.

### 2. Tier the planner LLM call — only run when heuristic *can't* decide

**Где:** `bookChatService.ts:3329-3454`
**Effort:** S

**Mechanism:** Каждый turn платит planner LLM call (Lite, ≤2k tokens, ~₽0.005). Для pure greeting/small-talk существующая `classifyBookChatQuestion` heuristic at 3106-3137 надёжна (length<30 + greeting tokens). Добавить fast pre-classifier:

**Impact:** ~10-15% turns — small-talk per typical chat traces. Saves ~₽0.005 × 0.12 × turns/day. Modest, но stacks с #3 ниже. Latency win на тех турнах: -200-400ms (один LLM round trip).

## T8: Self-hosted reranker (потенциально полный win)

**Дата записи:** 2026-05-01 (после Yandex Foundation experiment, который закрыли как невыгодный — но навёл на reranker tangent)

### Текущая ситуация

Прод использует Vertex Ranking API (`semantic-ranker-default@latest`) на каждый `search_paragraphs` и `search_scenes` вызов. На сложный chat-turn — ~3 rerank-вызовов. Это значит:

- Каждый rerank делает HTTP к US (`discoveryengine.googleapis.com`), отправляя evidence-paragraphs (фрагменты загруженных пользователями книг)
- Latency: 170-630ms на запрос (по логам tool runs)
- Стоимость: ~$0.004 за 1000 запросов = ~390 ₽/мес при текущем объёме (~1 000 ranks/мес)
- Привязка к `latest` — Google может тихо обновить модель и поломать precision на нашем golden eval'е

### Предложение

Развернуть `BAAI/bge-reranker-v2-m3` (568M параметров, MIT, multilingual, особенно сильна на русском) на отдельной CPU VPS:

- **Формат**: ONNX INT8 quantized (~600 MB RAM, ~80-150 ms на batch 30 docs)
- **Хостинг**: 2 vCPU + 4 GB RAM Timeweb-class VPS, ~700 ₽/мес
- **API**: внутренний FastAPI сервис в той же DC, что и web/worker (тот же провайдер)
- **Wire-up**: env-switch `RERANK_PROVIDER=vertex|self_hosted` в `vertexClient.rankRecords()`

### Экономика

Точка безубыточности — ~1 800 ranks/мес (≈ 600 chat-turn'ов/мес при 3 ranks/turn).

| Volume в месяц | Vertex Ranking ₽ | Self-host (700 ₽ VPS) | Δ |
|---|---|---|---|
| 1 000 ranks (~текущее) | ~390 | 700 | −310 (хуже) |
| 5 000 ranks | ~1 950 | 700 | +1 250 |
| 30 000 ranks | ~11 800 | 700 | +11 100 |
| 100 000 ranks | ~39 000 | 700 | +38 300 |

При активной нагрузке экономия ускоряется линейно — VPS-стоимость не растёт.

### Latency

| | Avg | p95 |
|---|---|---|
| Vertex Ranking (сейчас) | ~300 ms | ~630 ms |
| Self-hosted (INT8 на dedicated VPS) | ~140 ms | ~280 ms |

Чистая экономия latency на сложный turn (3 reranks): **−0.5...−1.1 сек**, заметная UX-разница.

### Что закрывается параллельно

1. **Один из двух trans-border каналов** — фрагменты книг больше не уезжают в Google US. В privacy policy упрощается раскрытие, для РКН-документов одна строчка вместо двух. Остаётся только LLM-канал (главный).
2. **Контроль модели** — `semantic-ranker-default@latest` может быть обновлён Google'ом без анонса. Self-hosted версия pinned, регрессии исключены.
3. **Future-proof для fine-tune'а** — открытая модель, можно дообучить на нашем литературном корпусе, если соберём датасет золотых пар query→relevant_paragraph (отдельный трек).

### План реализации (когда возьмёмся)

1. **Новый сервис** `apps/reranker/` — Python + FastAPI + ONNX Runtime + bge-reranker-v2-m3 INT8. Pre-load модели в Dockerfile. Эндпоинт `/rerank` совместим по контракту с текущим `VertexRankingRequest` / `VertexRankingResponseRecord`.
2. **`docker-compose.yml`** — добавить контейнер `reranker` (для local dev). Прод — отдельная VPS через deploy-скрипт.
3. **`packages/ai/src/vertexClient.ts`** — env-switch `RERANK_PROVIDER` в функции `rankRecords()`. Если `self_hosted` — HTTP к `RERANK_BASE_URL` (по умолчанию internal docker network в local, прод-URL VPS на проде).
4. **`.env.docker.example`** + `scripts/deploy/.env.production.example` — задокументировать новые env'ы с дефолтами на vertex (no breaking changes).
5. **Smoke-test** `npm run reranker:smoke` — проверка что сервис отвечает + dimensions совпадают.
6. **A/B на golden eval** — запустить eval с обоими провайдерами, сравнить precision (recall@5/10, MRR), latency, cost. Если bge проигрывает Vertex'у на >10% precision — закрываем тему через env-флаг.
7. **Деплой на VPS** — отдельный compose, pin'ed image версия.

**Объём работы:** 1-2 рабочих дня + сутки прогрева на A/B eval.

### Риски

- **Resource contention** — если class B (отдельная VPS) — нет. Если class A (на воркере) — возможны спайки latency при одновременном анализе книг и rerank'ах.
- **Quality regression** — bge-reranker-v2-m3 vs Vertex semantic-ranker. На бенчах MTEB они близкие; на нашем литературном корпусе нужна A/B-проверка перед катом.
- **Single point of failure** — если VPS падает, rerank ломается. Митигация: fallback в env-флаге обратно на Vertex Ranking (5 минут на переключение).

### Триггеры запуска

Делаем когда выполнится **хотя бы один из**:

1. Volume > **600 chat-turn'ов / месяц** (≈ 1 800 ranks/мес — точка безубыточности по деньгам)
2. Появляется enterprise-клиент, который требует «ничего в Google» из соображений compliance
3. Vertex Ranking ломает наш eval после очередного `latest` апдейта Google (регрессия в precision >5%)
4. Решаем дообучать reranker на собственном корпусе — тогда self-hosted становится единственным путём

### Что НЕ блокирует запуск

- Объём текущих 1 000 ranks/мес — не блокирует, но ROI отрицательный пока volume не вырастет
- Latency — текущий Vertex вариант приемлем для UX, self-host даёт улучшение, но не fix критичной боли
- Trans-border posture — улучшается, но это не блокер по 152-ФЗ (мы уже корректно disclosed Google как процессора)
