# Product / UX / Infra backlog

Сюда складываем будущие идеи, баги и launch-readiness задачи. RAG-specific
пункты живут отдельно в `docs/research/rag-audit-2026-04-30.md`.

Формат: одна задача = один заголовок третьего уровня с датой записи.
Когда берём в работу — снимаем отсюда и открываем PR/issue.

Приоритеты:
- **Release Blocker** — нельзя пускать публичный прод без решения.
- **Launch MVP** — желательно закрыть к запуску, но можно осознанно отложить.
- **Quick Fix** — маленький заметный баг/полировка.
- **Ops / Security** — эксплуатация, безопасность, стоимость, восстановление.
- **Growth / Catalog** — каталог, SEO, органика, маркетинг.
- **Later** — важно, но не должно блокировать первый запуск.
- **Decision Needed** — сначала нужно принять продуктовое/операционное решение.

## ✅ Shipped recently

- **2026-05-03 — Drop history retention cap from Free tier.** Изначально
  заложили `historyRetentionDays: 14` на Free, но это anti-pattern UX:
  storage cost ничтожен (~600 KB/юзер/год при 40 turns/мес), удалять
  чаты у бесплатных юзеров — повод для злости, не для апгрейда. Free
  и Plus теперь оба держат историю бессрочно. Убрано из
  `lib/tiers.ts` + `bucketUsage.ts` + `Profile.tsx`.

- **2026-05-03 — Tighten chat planner: tier router + adaptive reasoning + adaptive tool budget.**
  TOP-пункт RAG-аудита закрыт. Двухосевое решение планнера: `modelTier`
  (lite/pro) × `reasoningLevel` (minimal/low/medium). Расширены Pro-классы
  (retrospective reveal + emotional subtext) с лексическими триггерами по
  формулировкам. Tool budget теперь зависит от reasoningLevel:
  pro=6, lite+low=5, lite+minimal=3 — это убирает галлюцинацию anchor IDs
  на сложных Lite-вопросах (была root-причина: Lite не успевал собрать
  evidence за 3 поиска и подкидывал ID наугад).

  Эксперимент на 4 конфигурациях (BEFORE all-Pro / AFTER-v1 router-only /
  AFTER-v2 full / B-only adaptive-tools-only), 9 вопросов HP-Azkaban,
  ручная оценка по фактам (70%) + evidence (30%):
  - **AFTER-v2 (full): 9.42/10, $0.272 / 10 turns** ← выбран
  - BEFORE all-Pro: 9.36/10, $0.317 / 10 turns
  - B-only adaptive-tools: 9.00/10, $0.069 / 10 turns
  - AFTER-v1 router-only: 8.95/10, $0.065 / 10 turns

  AFTER-v2 строго доминирует BEFORE: лучше качество AND дешевле. Pareto
  improvement, не trade-off. См. `bookChatService.ts:3450-3520` (планнер
  prompt) и `bookChatService.ts:486-503` (`resolveMaxToolExecutionsForDecision`).

- **2026-05-02 — Per-user SSE event channel (`feat/sse-event-channel`).**
  Заменил inline chat streaming + 4-секундный поллинг анализа на единый
  `/api/events/stream` через Postgres LISTEN/NOTIFY. Закрыты:
  - Баг «loader чата появляется в другом чате» (Quick Fix) — стриминг-стейт
    теперь привязан к `sessionId`, лоадер светится только в той сессии где
    послал.
  - «Chat SSE сохранять между навигацией» (Later/Product Debt) — **page
    reload** во время генерации теперь подхватывает накопленный текст через
    `chat.snapshot` и продолжает в реалтайме. SPA-навигация между разделами
    (без F5) пока теряет partial text — оставил как мини-задачу в самом
    низу backlog'а, лечится подъёмом `streamingSessions` в Provider.
  - Поллинг `listAnalyzingBooks` каждые 4 секунды убран; UI обновляется
    через `book.analysis.done` NOTIFY от worker'а в течение долей секунды.
  Документация — `docs/research/sse-event-channel.md`.
- **2026-05-02 — Кириллица в имени файла при загрузке (`fix/multipart-cyrillic-filename`).**
  Регрессия от PR #016d880 «streaming multipart uploads»: ASCII-only
  sanitizer схлопывал «Гарри Поттер и Узник Азкабана.fb2» в «fb2» и upload
  падал с 415 «Unsupported format». Перешёл на Unicode-aware regex
  (`\p{L}\p{N}`), добавил регрессионный тест.

### Расхождение legal-доков с реальной архитектурой: реранкинг

**Записано:** 2026-05-01

**Проблема:** Legal-документы редакции `v1.2-2026-05-01` декларируют, что
переранжирование результатов поиска выполняется локально в РФ и не входит в
трансграничную передачу. Реальный код использует Vertex Ranking API
(`discoveryengine.googleapis.com`, US).

**Почему это blocker:** При публичной публикации legal-доков появится
письменное утверждение, которое архитектура сейчас опровергает.

**Опции закрытия:**
1. Развернуть свой reranker на отдельной VPS в РФ.
2. Переписать legal-доки так, что rerank тоже идёт через Google Vertex US.
3. Временно отключить Vertex Ranking (`VERTEX_RANKING_ENABLED=false`).

**Эксперимент по опции 1 проведён 2026-05-02 — свёрнут как преждевременный.**

Развернули `bge-reranker-v2-m3` (INT8 ONNX) на Timeweb VPS 4×3.3 ГГц QEMU
virtual / 4 GB / 80 GB за 1800 ₽/мес. FastAPI + onnxruntime, systemd unit,
доступ только из приватной сети по jump-host. Smoke на en/ru подтвердил что
модель работает корректно.

A/B провалилось по latency. На QEMU virtual CPU **отсутствует AVX-512/VNNI**,
поэтому INT8 на CPU не даёт ускорения. Реальные цифры:
- Vertex Ranking: avg 586 ms, p50 500 ms, p95 1190 ms (369 живых вызовов
  в продовой истории).
- Self-hosted на этом железе: ~20 секунд на batch 30 docs, иногда хуже
  при параллельных запросах (накопительная очередь).

В результате каждый rerank-запрос упирался в `RERANK_TIMEOUT_MS=5000` →
catch в `rankRecords()` → fallback на pre-rerank scores. Чат отвечал, но
эффективно без rerank'а.

**Экономика на текущем volume не оправдывает self-host:**
- Сейчас (~22 turn/день, мы на стенде): Vertex ~$4/мес ≈ 450 ₽.
- Закрытая beta (10-50 users): Vertex ~600 ₽/мес.
- Free beta (200-500 users): Vertex ~2 000 ₽/мес.
- Break-even с High CPU VPS 5 390 ₽/мес — **~300 turn/день**, что
  соответствует ~600 active users в месяц.

**Что вернёт эксперимент в работу:**
1. Volume стабильно превысит ~300 turn/день, ИЛИ
2. Появится compliance-клиент, требующий «всё в РФ» как hard requirement, ИЛИ
3. Vertex Ranking сломает наш golden eval после очередного `latest`-апдейта
   (регрессия precision >5%).

При повторе нужно сразу брать **High CPU 4×5 ГГц** (5 390 ₽/мес) или
**Dedicated CPU** с реальным процессором — на cheap QEMU bge-reranker не
тянет. Подробности и план реализации — `docs/research/rag-audit-2026-04-30.md`
секция T8 (план остаётся валидным, упирается только в железо).

**Решение по legal-блокеру до пересмотра:** опция 2 — переписать
privacy §6/§7 так, чтобы они отражали фактическую архитектуру (rerank
через Vertex AI US). Это технически не новый канал передачи: фрагменты
параграфов и так уезжают в Vertex Gemini для генерации ответа, rerank
ходит туда же.

### Tariff UX: баннеры приближения и достижения лимита

**Записано:** 2026-05-03

**Проблема:** После того как тарифы запущены (PR `feat(billing)`), у юзера
нет визуальных сигналов **до** того как он упрётся в стену. 429 в чате /
upload приходит "вдруг". Pro→Lite soft-fallback тоже не сообщается явно
— юзер видит, что ответ стал проще, но не понимает почему.

**Ожидаемо:** Три состояния в UI:

1. **Approaching** (≥80% bucket'а использовано):
   - Тонкий неинвазивный баннер в шапке чата или в counter-badge:
     «Pro: 60 / 75 — почти исчерпан»
   - Цвет — warning, не ошибка
   - CTA «Перейти на Plus +» (для Free) или просто info
2. **Soft fallback** (Pro=0 у Plus, чат продолжает на Lite):
   - Status-badge в ответе ассистента: "Ответ на Lite — Pro лимит
     исчерпан, обновится X числа"
   - Не блокирует, не пугает — просто прозрачно сообщает
3. **Hard limit** (Lite=0 для Free, или analyses=0 для Plus):
   - Чат: модалка/inline-message «Достигнут лимит ответов в этом
     периоде. Обновится X числа.» + CTA «Перейти на Plus»
   - Upload: PaywallModal (уже есть) с обновлёнными copy
   - Library save (Free 5/5): inline-tooltip + CTA

Источник данных — `/api/me/usage` (уже работает). UI-компонент —
переиспользуемый `UsageBadge` или `LimitWarning` в чат-шапке.

### Manual tier selector в чате (Авто / Lite / Pro)

**Записано:** 2026-05-03

**Проблема:** Сейчас планнер сам решает Lite/Pro по тексту вопроса.
Это работает в большинстве случаев, но юзер не имеет контроля. Если
он задаёт сложный analytical вопрос, который планнер не зацепит как
"retrospective reveal" — пойдёт на Lite, ответ будет беднее.

Поскольку мы вводим Plus с Pro-buckets, естественно дать пользователю
явный выбор — иначе Plus-тариф воспринимается как "плачу за ту же
автоматику, только дороже".

**Ожидаемо:** Селектор у поля ввода чата (segmented control или
dropdown):

```
[ Авто ▼ ]   ← модель сама выбирает (default)
[ Lite ]     ← быстрый дешёвый ответ, simple-fact формат
[ Pro ]      ← глубокий разбор, медленнее, тратит Pro-бюджет
```

Поведение:
- Default: Авто (planner решает) — текущая логика
- Manual Lite: forceLiteTier=true (re-use existing override path)
- Manual Pro: forceProTier=true (новый путь — overrides planner's
  Lite decision when budget allows)
- Manual Pro для Free / Plus с Pro=0 → не показываем кнопку "Pro" или
  показываем locked с tooltip "Доступно на Plus / лимит исчерпан"
- Selector состояние per-thread, не global — юзер может в одном
  чате выбрать Pro, в другом оставить Авто
- Сохранять как preference: thread-level в `BookChatThread.preferredTier`

Edge-case: что если юзер выбрал Pro вручную, а Pro-бюджет 0?
Вариант A: блок отправки + tooltip "Pro лимит исчерпан, переключитесь
на Lite или Авто". Вариант B: автоматический fallback с warning
("отправили на Lite, потому что Pro исчерпан"). **Я бы взял A** —
явный выбор юзера должен уважаться, а не молча подменяться.

ROI: значительно повышает воспринимаемую ценность Plus (юзер
контролирует), и снимает с планнера нагрузку быть всегда правым.

### History compaction: эксперимент 4→8 пар + context-fill indicator + manual compact

**Записано:** 2026-05-03

**Проблема:** Сейчас `ensureCompactedHistory` сжимает старые turn'ы
в lite-summary при достижении ~4 пар (см. `bookChatService.ts`).
Это работает, но:
1. Слишком ранняя компакция мешает Vertex implicit cache —
   системный + первые turn'ы остаются стабильными, могли бы
   кэшироваться, но мы их уже сжали.
2. Юзер не видит **сколько** контекста занято и когда оно
   рестартует. В «нормальных» AI-чатах (Claude UI, ChatGPT) есть
   visible indicator заполненности окна.
3. Нет опции **ручной** компакции — иногда юзер хочет начать
   заново после длинной ветки, но не терять весь чат.

**Эксперимент 1: повысить hysteresis с 4 до 8 пар.**
Гипотеза — более длинная история = больше cache hit'ов в
последующих turn'ах. Сейчас при cache hit rate 22% на Pro и 9%
на Lite (пост-сегодняшнее измерение). Возможно при 8 парах cache
дойдёт до 35-40% за счёт более стабильного prefix'а.

Риск: длиннее prompt → больше input tokens на каждый turn до
компакции. Cost-нейтрально только если cache реально срабатывает.
Замеряется через `BookChatTurnMetric.modelCachedInputTokens`
после A/B (групп юзеров с разной hysteresis).

**Эксперимент 2: visible context-fill indicator.**
Tiny progress-bar в шапке чата:

```
[████████░░] 80%   ← скоро автоматическая компакция
```

При hover/click открывается panel:
- «Использовано: 8k / 32k токенов»
- Кнопка «Сжать сейчас» (force compact even if под порогом)
- «Начать новый чат» (создать новый thread, текущий остаётся)
- Toggle «Сжимать автоматически: вкл/выкл» (если выкл — юзер
  сам решает когда жать)

Это выводит chat в категорию **expert-tool**, а не black-box.
Пользователю с длинными research-сессиями это критично.

**Эксперимент 3: cost-attribution per compaction.**
Каждая компакция тратит lite-LLM call (~$0.001-0.002). Сейчас
эти затраты теряются в общей `embeddingCostUsd`/`chatCostUsd`.
Нужно логировать в `BookChatThread.compactionCostUsd` или
аналогичное — для unit-econ visibility.

ROI:
- Эксп 1 (4→8) — потенциально −15-20% LLM cost через cache.
- Эксп 2 (UI) — повышает воспринимаемое качество продукта,
  attracts power-users.
- Эксп 3 (attribution) — closes metering gap.

Все три можно делать независимо. Триггер: после стабилизации
тарифов и сбора первого месяца use-data.

### Чат: citation precision (cite-to-thesis exact match)

**Записано:** 2026-05-03 (вышло из эксперимента planner-tightening)

**Проблема:** Даже у лучшей конфигурации (AFTER-v2) evidence-балл 8.65/10.
Модель цитирует **соседние** параграфы вместо точных, например:
- Тезис «дементор в поезде» — модель ставит ch5:p259 (это сцена пира
  Дамблдора, не дементор), хотя точный source — ch5:p194 («Дементор. —
  Люпин раздавал шоколад»).
- Тезис «имя Клювокрыл» — модель ставит ch16:p160 вместо ch16:p165
  где имя реально звучит.
- Тезис «официальная версия о Сириусе» — ставит ch10:p224, ch10:p228,
  хотя ближе ch10:p225, ch10:p229, ch10:p231.

Это не галлюцинация ID (параграфы реальные, главы правильные), это
смещение на 1-3 параграфа от точного места. Скорее всего модель
выбирает первый параграф из evidence-группы вместо параграфа с
точной фразой.

**Ожидаемо:** Несколько направлений:
1. Усилить system prompt: «выбирай параграф где буквально звучит
   ключевое слово из тезиса, не первый из найденной группы».
2. В evidence pack маркировать «keyword anchor» — параграф где
   нашёлся буквальный термин из query — отдельным тегом.
