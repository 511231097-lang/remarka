# Product / UX / Infra backlog

Сюда складываем будущие идеи, баги и launch-readiness задачи. RAG-specific
пункты живут отдельно в `docs/research/rag-audit-2026-04-30.md`.

Формат: одна задача = один заголовок третьего уровня с датой записи.
Когда берём в работу — снимаем отсюда и открываем PR/issue.

Приоритеты:
- **Release Blocker** — нельзя пускать публичный прод без решения.
- **Launch MVP** — желательно закрыть к запуску, но можно осознанно отложить.
- **Quick Fix** — маленький заметный баг/полировка.
- **Ops / Security** — эксплуатация, безопасность, стоимость, восстановление.
- **Growth / Catalog** — каталог, SEO, органика, маркетинг.
- **Later** — важно, но не должно блокировать первый запуск.
- **Decision Needed** — сначала нужно принять продуктовое/операционное решение.

## 🧠 DECISIONS NEEDED

### Определить точный launch scope

**Записано:** 2026-05-01

**Проблема:** Сейчас backlog смешивает задачи для закрытого стенда, публичной
free beta, SEO-запуска и платного запуска. Из-за этого непонятно, что реально
блокирует ближайший релиз.

**Ожидаемо:** Явно зафиксировать формат первого запуска: free beta или сразу
платный продукт, нужен ли SEO-трафик на старте, сколько книг в каталоге, какие
фичи доступны и что сознательно out of scope.

---

### Почта: abuse@, hello@, privacy@ и алиасы

**Записано:** 2026-05-01

**Проблема:** Нужно понять, нужны ли отдельные адреса `abuse@`, `hello@`,
`privacy@`, или часть адресов можно объединить/сделать алиасами.

**Ожидаемо:** Решение по адресам согласовано с legal/операционными
процессами. После решения — подключить почтовика и реализовать алиасы, чтобы
адреса из legal-документов реально принимали письма.

---

### CDN и доступность в РФ

**Записано:** 2026-05-01

**Проблема:** Нужно понять стратегию CDN. Cloudflare может быть нестабилен
или недоступен для части пользователей в РФ, поэтому нельзя молча завязаться
на него без fallback-плана.

**Ожидаемо:** Исследовать варианты: без CDN на старте, российский CDN,
S3/Timeweb-совместимая раздача, nginx caching, fallback для статики и
обложек. Зафиксировать выбранную схему и риски.

---

### Авторизация: форма vs отдельная страница

**Записано:** 2026-05-02

**Проблема:** Сейчас `/signin` — это отдельная страница, на которую
редиректит middleware/protected-layout при попытке зайти в auth-only
раздел. Это уводит пользователя из контекста (был на карточке книги —
оказался на полностраничной форме), теряет state каталога, плюс лишний
переход между «решил войти» и «вернулся к книге».

**Ожидаемо:** Принять решение по форме входа: модалка с overlay поверх
текущей страницы, drawer, inline-блок при клике на «Войти», или сохранить
отдельную страницу. Учесть SmartCaptcha (виджет должен жить в DOM формы),
consent-чекбокс, callbackUrl, корректную работу при auth-redirect от
NextAuth и SSR совместимость.

## 🔴 RELEASE BLOCKERS

### Домен, финальное название и брендовая замена

**Записано:** 2026-05-01

**Проблема:** Нужно окончательно выбрать домен и точное название продукта,
заменить все вхождения рабочего названия по проекту, а также убрать заглушки

**Почему это blocker:** Публичный запуск нельзя делать с временным именем,
заглушками Ф.И.О./ИНН или неверными OAuth/payment/email/legal настройками.

**Exit criteria:** Финальное название и домен утверждены, все публичные
тексты/metadata/legal/footer/export/env/OAuth redirect URI приведены к ним.

### Расхождение legal-доков с реальной архитектурой: реранкинг

**Записано:** 2026-05-01

**Проблема:** Legal-документы редакции `v1.2-2026-05-01` декларируют, что
переранжирование результатов поиска выполняется локально в РФ и не входит в
трансграничную передачу. Реальный код использует Vertex Ranking API
(`discoveryengine.googleapis.com`, US).

**Почему это blocker:** При публичной публикации legal-доков появится
письменное утверждение, которое архитектура сейчас опровергает.

**Опции закрытия:**
1. Развернуть свой reranker на отдельной VPS в РФ.
2. Переписать legal-доки так, что rerank тоже идёт через Google Vertex US.
3. Временно отключить Vertex Ranking (`VERTEX_RANKING_ENABLED=false`).

### Баг: loader чата появляется в другом чате

**Записано:** 2026-05-01

### Баг: не подсвечиваются ссылки вида `[ch3:p1, ch6:p177]`

**Записано:** 2026-05-01

**Проблема:** Модель периодически возвращает ссылки на источники в формате
`[ch3:p1, ch6:p177, ch6:p186]`, но интерфейс их не подсвечивает. Нужно
разобраться, это неподдержанный формат citation parser'а или модель
галлюцинирует номера параграфов, которых нет в retrieved context.

**Ожидаемо:** Понять source of truth для ссылок на главы/параграфы, покрыть
parser нужными форматами или запретить модели такой формат, если он не может
быть валидно привязан к тексту. Невалидные ссылки не должны выглядеть как
рабочие цитаты.

### Чат: сворачивать большие сообщения пользователя

**Записано:** 2026-05-01

**Проблема:** Если пользователь присылает большое сообщение, оно занимает
слишком много места в чате и ломает читаемость истории.

**Ожидаемо:** Длинные пользовательские сообщения красиво сворачиваются:
preview, кнопка “Показать ещё” / “Свернуть”, без скачков layout и с нормальным
поведением на mobile.

---

### Тема: auto по умолчанию + убрать FOUC при тёмной теме

**Записано:** 2026-05-02

**Проблема:** Два связанных недочёта оформления темы:
1. Если пользователь не выбирал тему явно, активной должна быть `auto`
   (следовать `prefers-color-scheme`), а не светлая по умолчанию.
2. Если тема явно выбрана как тёмная, при перезагрузке страницы первые
   ~100–300 мс рендерится светлая тема (FOUC), потом переключается на
   тёмную. Это бликует и выдаёт hydration-mismatch паттерн.

**Ожидаемо:** Дефолт `auto`, тема применяется до первого пэйнта (обычно
через blocking inline-script в `<head>`, который читает localStorage и
выставляет `data-theme` на `<html>`), без видимой вспышки светлой темы
для пользователей с тёмной системной/выбранной темой.

---

### Скрыть auth-only UI-элементы для анонимов

**Записано:** 2026-05-02

**Проблема:** После выкатки публичного обзора каталога/книг анонимный
пользователь видит элементы интерфейса, которые ему ничего не дают:
кнопка «В библиотеку» на карточках каталога, пункт «Мои книги» в

**Ожидаемо:** Эти элементы скрыты для не-авторизованных. Точки входа
для анонима — «Войти» (или аналог) и публичный обзор. Авторизационные
действия не маячат как доступные кнопки, которые ничего не делают или
дают ошибку.

### Upload security и обработка опасных файлов

**Записано:** 2026-05-01

**Проблема:** Upload принимает EPUB/FB2/PDF/ZIP и парсит их на сервере. Нужно

**Ожидаемо:** Лимиты по размеру файла, распакованному размеру, страницам/

### Security review API и CSRF/abuse matrix

**Записано:** 2026-05-01

**Проблема:** Middleware пропускает все `/api/*`, а авторизация проверяется
внутри отдельных route handlers. Перед запуском нужно пройти все API и
составить матрицу: public/auth/admin/internal, mutating/read-only,
rate-limited/not, CSRF-sensitive/not, expensive/not.

**Ожидаемо:** Все мутации защищены ожидаемым auth/admin/internal token,
дорогие endpoints имеют rate limits/quotas, internal API нельзя вызвать
снаружи без секрета, CSRF-риски явно закрыты или приняты.

### Cost guardrails для LLM/анализа

**Записано:** 2026-05-01

**Проблема:** Чат, анализ, showcase и rerank могут быстро жечь деньги.
Нужны технические лимиты: per-user/per-day quotas, spend caps, concurrency
caps, kill-switches, degraded mode и видимость стоимости.

**Ожидаемо:** Нельзя случайно потратить большой бюджет из-за одного
пользователя, багнутого job'а или атаки. Админка/логи показывают расход по
пользователю, книге, модели и фиче.

### SSR для каталога и обзора публичных книг

**Записано:** 2026-05-02

**Проблема:** `/explore` и `/book/[id]` сейчас Client Components — данные
грузятся через клиентский `fetch` после гидратации. Поисковик получает
пустой HTML без контента, не индексирует названия, авторов, summary,
характеристики книг. SEO-блок (метаданные, OG, sitemap) бесполезен без
этого фундамента: даже если sitemap отдаст книгу, бот её увидит пустой.

**Ожидаемо:** Каталог и обзор книги переведены на SSR (Server Component
с прямым обращением к БД либо `fetch` на свои API в server-runtime),
HTML отдаётся уже наполненным. Метаданные (`generateMetadata`) тянут
title/description/og-image из той же серверной выборки. Клиентские
кнопки (фильтры, пагинация, «В библиотеку» и т.п.) остаются hydrated
поверх серверного HTML. Этот пункт — предусловие для большинства задач
из SEO-блока.
