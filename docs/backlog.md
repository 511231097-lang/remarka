### Расхождение legal-доков с реальной архитектурой: реранкинг

**Записано:** 2026-05-01

**Проблема:** Legal-документы редакции `v1.2-2026-05-01` (privacy п.6 (д) и п.7)
декларируют, что **переранжирование результатов поиска выполняется локально
на инфраструктуре в РФ** и в трансграничную передачу не входит. **Это пока
не соответствует реальности** — продакшн-код использует Vertex Ranking API
(`semantic-ranker-default@latest` через `discoveryengine.googleapis.com`,
US). Каждый rerank-вызов отправляет фрагменты загруженных пользователями
файлов в Google US.

**Почему это release-blocker:**
- При публикации legal-документов на публичной странице появится письменное
  утверждение, что rerank делается в РФ.
- Реальный сетевой трафик утверждение опровергает.
- Это создаёт risk: при проверке РКН / иске пользователя / запросе
  правообладателя — мы предъявляем policy в одном виде, а архитектура
  показывает другое. Это **выглядит как заведомо ложная декларация**, что
  значительно хуже чем «честно раскрыли что Vertex Ranking тоже trans-border».

**Опции:**
1. **Реализовать self-hosted reranker** (план T8 в `rag-audit-2026-04-30.md`):
   развернуть `BAAI/bge-reranker-v2-m3` на отдельной CPU VPS в РФ. Truth ←
   соответствует policy. **Рекомендуемый путь.** Явная целевая формулировка:
   свой реранкер на отдельной VPS.
2. **Откатить декларацию в legal-доках** — переписать privacy п.6 (д) и п.7
   так, что rerank также через Google Vertex (US) и входит в trans-border.
   Truth ↔ соответствует, но мы теряем «легально-чистую посадку» которую
   юр-чат специально отстраивал.
3. **Временно отключить Vertex Ranking** (`VERTEX_RANKING_ENABLED=false`):
   policy остаётся как есть, реальность временно соответствует. Но мы
   теряем ~15-20% retrieval precision по golden eval. Опасный для UX
   trade-off.

**Триггер закрытия release-blocker'а:** один из трёх путей выше реализован.
До этого — **не публиковать legal-документы редакции v1.2 на публичной
странице сайта**. Файлы могут быть в репо, но прод-сайт `/legal/*` на
публичном домене должен либо показывать предыдущую редакцию (v1.1), либо
быть скрыт от индексации.

### Chat SSE сохранять между навигацией

**Записано:** 2026-05-01

**Проблема:** SSE-стрим ответа ассистента сбрасывается, если пользователь
переходит в другой чат, в другую книгу, или вообще в другой раздел сайта.
Ответ обрывается, прогресс теряется.

**В идеале:** стрим продолжает идти в фоне; когда пользователь возвращается
в этот чат — видит готовый ответ (или продолжение прогресса). Поведение
как в ChatGPT/Claude UI.

**Подсказки для будущей реализации:**
- Текущий стрим — `/api/books/[bookId]/chat/sessions/[sessionId]/stream` через
  fetch + reader в `BookChat.tsx`. Стрим привязан к React component lifecycle:
  unmount → reader cancelled.
- Варианты: (а) поднять streaming-state выше дерева компонентов
  (ChatContext / zustand / store), чтобы стрим жил пока пользователь в
  пределах /book/[bookId]; (б) сервис-воркер как persistent-host для стрима;
  (в) серверная буферизация ответа + возможность re-connect к streaming-id'у.
- Самый простой первый ход — вариант (а) для перехода между чатами одной
  книги. Полное (б)+(в) — для переходов между разделами / закрытий вкладки.

### Блокировка книги по жалобе правообладателя — UX для пользователя

**Записано:** 2026-05-01

**Проблема:** Когда жалоба правообладателя принимается (CopyrightComplaint
переходит в `status=accepted` в админке), книга должна быть ограничена в
доступе у её владельца. Но сейчас механизма блокировки на уровне `Book`
вообще нет, а если он появится — пользователь увидит просто что «книга
пропала / не открывается» без объяснения. Это плохо и юридически
(пользователь имеет право знать причину и подать встречное обращение
по ст. 1253.1 ГК РФ), и операционно (юзер пишет в support — мы тратим
время на ручное объяснение).

**В идеале:** При попытке открыть заблокированную книгу — explicit-экран:

> «Эта книга временно ограничена по жалобе правообладателя.
> Если вы считаете блокировку необоснованной — вы можете направить
> встречное обращение на abuse@remarka.app или через [форму].»

С кнопкой на counter-claim flow и ссылкой на текст «Жалоба правообладателя»
в legal-документах, где порядок описан.

**Контекст:**
- Юридическая база — ст. 1253.1 ГК РФ + п.5 «Условий загрузки произведения»
  + страница «Жалоба правообладателя» (`/legal/copyright`).
- В БД уже есть статус `CopyrightComplaint.status = 'counter_received'` для
  встречных обращений — частично готов модель.
- В админке `/admin/copyright-complaints/[id]` можно сменить статус на
  `accepted`, но никакого автодействия на Book пока нет.

**Подсказки для будущей реализации:**
- Связать `CopyrightComplaint` с конкретной `Book`: добавить
  `disputedBookId: String?` колонку (или хранить как структурированный JSON
  в `disputedUrls`). Админ при разборе жалобы должен мочь одним кликом
  найти спорную книгу и привязать.
- На `Book` добавить:
  - `blockedAt: DateTime?`
  - `blockedByCopyrightComplaintId: String?` (FK)
  - `blockReason: String?` (для возможных будущих типов блокировки помимо
    copyright — пока только `'copyright_complaint'`)
- В `bookChatService` и retrieval — early return с blocked-флагом, ассистент
  не должен отвечать на вопросы о заблокированной книге.
- В `BookChat.tsx`, `BookCard`, `library/explore` — single source of truth
  на блокировку, показ explicit-сообщения вместо контента.
- Counter-claim flow — переиспользовать существующую `CopyrightDoc` форму,
  но в режиме `claimantType=counter_claimant` (новый вариант enum'а), с
  привязкой к существующей жалобе (`relatedComplaintId`).
- При `accept` в админке — отправить пользователю email-уведомление о
  блокировке + ссылку на counter-claim. Email в backlog'е через ту же
  ветку что вообще SMTP-настройка для уведомлений на abuse@.
